**关键设计:**

- 向量库(Milvus)只存 `vector + mysql_id`，不存业务数据

- 搜索时先查 Milvus 得 ID，再回 MySQL 取详情

## 二、关键技术问题回应

### 2.1 107 万封面生成向量耗时

下面是耗时预估，在使用 cpu 处理的情况下，百万级的图片处理总共耗时一定是在月级别的，使用 **GPU** 的话，速度会缩短到周级别，视硬件规格而定。

<table>
<tr>
<td>配置<br/></td><td>速度<br/></td><td>总耗时<br/></td></tr>
<tr>
<td>**单核CPU**<br/></td><td>~100张/小时<br/></td><td>**约446天**<br/></td></tr>
<tr>
<td>**8核并行**<br/></td><td>~800张/小时<br/></td><td>**约55天**<br/></td></tr>
<tr>
<td>**16核并行**<br/></td><td>~1600张/小时<br/></td><td>**约28天**<br/></td></tr>
</table>

**优化策略:**

- 分批处理：每批 1 万张，断点续传
- 支持中断恢复：已处理封面通过 `cover_hash` 自动跳过

### 2.2 Faiss/Milvus Index 类型

使用 Milvus HNSW 索引 (Faiss 不支持百万级数据量的处理，扩展能力弱):

```yaml
Index Type: HNSW (Hierarchical Navigable Small World)
Parameters:
  - M: 16                    # 每层最大连接数
  - efConstruction: 200       # 构建时搜索范围
  - ef: 64                    # 查询时搜索范围(可调)
  - metric_type: COSINE      # 余弦相似度
```

选择理由:

- 107 万数据量下查询延迟 < 50ms
- 内存占用适中 (约 2-3GB 索引)
- 支持增量插入，无需重建
- 精度 vs 速度平衡最优

### 2.5 增量更新机制 (10 万新书)

无需重建 107 万向量，仅增量插入:

```markdown
新增10万封面流程:
    │
    ▼
┌─────────────────┐
│ 1. 感知哈希去重  │ ← 与现有107万比对，秒级完成
│    (cover_hash)  │
└─────────────────┘
    │
    ▼
┌─────────────────┐
│ 2. Nomic向量化   │ ← 仅处理非重复封面
│    (~80ms/张)   │
└─────────────────┘
    │
    ▼
┌─────────────────┐
│ 3. Milvus插入    │ ← 增量插入，HNSW支持动态添加
│    (无需重建)   │
└─────────────────┘
    │
    ▼
┌─────────────────┐
│ 4. MySQL映射更新 │ ← 插入新记录
└─────────────────┘
```

### 2.6 完整查询流程 (单图搜索)

```yaml
用户上传新图
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 1: 预处理 (50ms)                   │
│ • 下载/解码 → 去噪 → 224×224            │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 2: 感知哈希 (5ms)                  │
│ • 快速查重，如完全重复直接返回           │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 3: Nomic向量化 (80ms)              │
│ • CPU生成768维向量                      │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 4: Milvus检索 (30ms)               │
│ • HNSW搜索Top-10                        │
│ • 返回相似度分数                        │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 5: 阈值分流                        │
│ • ≥0.95: 直接返回最佳匹配               │
│ • 0.85-0.95: 进入多模态验证             │
│ • <0.85: 返回"无匹配"                  │
└─────────────────────────────────────────┘
    │ (如需验证)
    ▼
┌─────────────────────────────────────────┐
│ Step 6: QwenVL验证 (1-2s)               │
│ • 比对原图与候选封面                     │
│ • AI判断"是否同一封面"                  │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│ Step 7: 组装结果                        │
│ • 查MySQL获取元数据(title/isbn等)      │
│ • 返回JSON                              │
└─────────────────────────────────────────┘
```

**总耗时预估:**

- 快速路径 (≥0.95): ~200ms
- 验证路径 (0.85-0.95): ~2-3s
- 无匹配 (<0.85): ~150ms

### 2.7 500 万封面性能预估

Milvus 可支持千万级向量的快速检索，500 万内单机部署的 Milvus 向量数据库完全可行，查询速度仍在 100ms 内。
