### 2.2 技术实现层面

#### 🔴 问题4：图片预处理细节不明确
**现状**：仅提到"格式转换 → 扫描版去噪 → 224×224标准化"

**项目实际情况：**
- ✅ **采用阿里云方案**（Hologres/通义千问）
- ✅ **图片已存储在 OSS 中**
- ✅ **图片大小均在 1M 以内**

**预处理策略（已确定）：**
- ✅ **保留基础验证**：格式验证、可读性检查
- ✅ **必要预处理**：感知哈希计算（用于去重）
- ✅ **委托服务端**：尺寸调整、归一化等由阿里云服务自动处理
- ❌ **无需复杂预处理**：去噪、格式转换等可省略（服务端自动处理）

**预处理流程（简化版）：**
```
OSS中的图片
    ↓
基础验证（格式、可读性）
    ↓
感知哈希计算（去重）
    ↓
调用向量化服务（Hologres/通义千问自动处理）
    ↓
获得向量
```

**建议细化**：
- **格式支持**：JPEG/PNG/WebP（阿里云服务支持的主流格式）
- **基础验证**：
  - 从 OSS 读取图片，验证格式和可读性
  - 记录损坏或无法读取的图片
- **感知哈希**：计算 cover_hash 用于快速去重
- **异常处理**：
  - OSS 图片访问失败的处理
  - 图片格式不支持时的错误提示
  - 向量化失败的重试机制

**详细说明**：详见 `图片预处理与向量计算说明.md`

### 4.4 阿里云基础设施调整（重要）
**如果采用阿里云全套基础设施，技术方案需要重大调整：**

详见单独文档：`阿里云基础设施调整方案.md`

**核心调整点：**
1. **向量化**：Nomic 本地部署 → Hologres AI Function（BGE模型）
2. **向量数据库**：Milvus → Hologres（实时数仓）
3. **多模态验证**：Qwen2.5-VL 本地部署 → 通义千问-VL API
4. **存储**：本地存储 → OSS（对象存储）
5. **数据库**：自建 MySQL → RDS MySQL
6. **服务器**：自建服务器 → ECS/函数计算

**优势：**
- 开发周期缩短 1周（30人天 → 24人天）
- 性能提升 4-8倍（GPU加速）
- 运维成本大幅降低
- 自动扩缩容，支持千万级数据

**注意事项：**
- 需要提前进行 POC 验证 Hologres AI Function
- 成本需要根据实际使用量评估
- 数据迁移方案需要详细设计

## 五、总结

该技术方案整体架构合理，技术选型恰当。针对**快速项目**的特点，建议重点关注：

### 核心必须项
1. **容错机制**：基础的重试和异常处理（Milvus/MySQL 连接失败处理）
2. **图片预处理细节**：明确处理流程，确保功能实现
3. **API 接口规范**：详细的请求/响应格式，便于对接
4. **部署方案**：清晰的部署步骤和环境要求

### 可暂缓项
- **监控日志**：使用 Python logging 记录关键信息即可，无需完整监控体系
- **安全性**：内网环境可简化，仅保留基础的输入验证
- **完整测试**：基础功能测试即可，无需完整的测试体系

### 建议
在保证核心功能实现的前提下，优先完成高优先级项，其他功能可在后续迭代中补充完善。
