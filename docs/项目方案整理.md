# 书籍封面向量检索系统 - 项目方案整理

## 一、项目概述

### 1.1 项目背景
构建一个基于向量检索的书籍封面相似性查找系统，支持通过图片快速检索相似的书籍封面。

### 1.2 核心需求
1. 从MySQL表中批量读取书籍信息（包含图片链接）
2. 对图片进行向量化处理，存入向量数据库
3. 同时提取图片中的文字信息（OCR）
4. 支持通过图片进行相似性检索
5. 检索结果支持OCR文字二次过滤

### 1.3 技术栈
- **数据源**：MySQL RDS（表名：book_info_table，可配置）
- **向量数据库**：Milvus（阿里云版）
- **向量化服务**：阿里云 Embedding API（通义千问多模态）
- **OCR服务**：阿里云OCR服务
- **图片存储**：OSS（通过cover_link字段存储链接）
- **处理服务**：阿里云ECS或函数计算

### 2.2 数据流转流程

**批量处理流程：**
1. 从MySQL表 `book_info_table` 分批查询数据
2. 遍历每行数据，获取 `cover_link` 字段
3. 从链接下载图片到内存
4. 对图片进行基础验证（格式、可读性）
5. 计算感知哈希（用于去重）
6. 并行调用两个API：
   - 阿里云Embedding API：获取图片向量
   - 阿里云OCR API：提取图片文字
7. 组装数据：向量 + 元数据 + OCR文字
8. 批量插入Milvus
9. 记录处理进度，支持断点续传

**检索流程：**
1. 接收查询图片（URL或Base64）
2. 调用Embedding API获取查询向量
3. 在Milvus中进行向量相似度检索（Top-K）
4. 可选：使用OCR关键词进行二次过滤
5. 返回匹配结果（包含相似度分数和元数据）

## 三、Milvus表结构设计

### 3.1 Collection Schema设计

| 字段名 | 类型 | 说明 | 用途 |
|--------|------|------|------|
| `id` | INT64 | 主键ID（Milvus自增） | 唯一标识 |
| `mysql_id` | INT64 | MySQL中的原始ID | 关联回MySQL |
| `sku` | VARCHAR(128) | 商品SKU | 业务标识 |
| `isbn` | VARCHAR(32) | ISBN号 | 业务标识 |
| `cover_link` | VARCHAR(512) | 图片链接 | 图片访问 |
| `cover_hash` | VARCHAR(16) | 感知哈希 | 快速去重 |
| `ocr_title` | VARCHAR(256) | 提取的书名 | 快速匹配 |
| `ocr_author` | VARCHAR(128) | 提取的作者 | 快速匹配 |
| `ocr_summary` | VARCHAR(500) | OCR文字摘要（前500字） | 关键词过滤 |
| `ocr_keywords` | VARCHAR(200) | 提取的关键词 | 关键词过滤 |
| `embedding` | FLOAT_VECTOR(768) | 图片向量 | 相似度检索 |

### 3.2 索引配置

- **索引类型**：HNSW（Hierarchical Navigable Small World）
- **参数配置**：
  - `M`: 16（每层最大连接数）
  - `efConstruction`: 200（构建时搜索范围）
  - `ef`: 64（查询时搜索范围，可调）
- **相似度计算**：COSINE（余弦相似度）
- **性能预期**：107万数据量下查询延迟 < 50ms

## 四、核心功能模块

### 4.1 批量处理模块

**功能：**
- 从MySQL分批读取数据
- 图片下载和验证
- 向量化和OCR并行处理
- 批量插入Milvus
- 断点续传支持

**关键设计：**
- **分批大小**：可配置（建议1000-10000条/批）
- **并发控制**：控制API调用并发数，避免限流
- **错误处理**：记录失败记录，支持重试
- **进度追踪**：记录已处理数量，支持中断恢复

### 4.2 图片预处理模块

**处理步骤：**
1. 从cover_link下载图片
2. 格式验证（JPEG/PNG/WebP）
3. 可读性验证（确保图片完整）
4. 计算感知哈希（用于快速去重）

**注意：**
- 图片大小已在1M以内，无需大小检查
- 尺寸调整、归一化等由阿里云Embedding API自动处理
- 无需复杂的去噪处理（快速项目可暂缓）

### 4.3 向量化模块

**实现方式：**
- 调用阿里云Embedding API（通义千问多模态模型）
- 传入图片URL或Base64编码
- 返回768维向量

**性能：**
- 处理时间：~10-20ms/张
- 支持批量调用（如果API支持）

### 4.4 OCR文字提取与处理模块

**实现方式：**
- 调用阿里云OCR服务
- 传入图片URL或Base64编码
- 提取图片中的完整文字内容

**文字处理：**
1. **文字清理**：去除特殊字符、空格规范化
2. **结构化提取**：
   - 书名提取（封面中心或上方，字体最大）
   - 作者提取（"作者："、"著："等关键词后）
   - 出版社提取（"出版社："、"出版："等关键词后）
   - ISBN提取（正则匹配ISBN格式）
3. **摘要生成**：提取前500字作为摘要
4. **关键词提取**：使用分词工具提取关键词

**存储策略（混合存储）：**
- **Milvus存储**：ocr_title, ocr_author, ocr_summary, ocr_keywords
- **MySQL存储**：完整OCR文字（可选，如果文字较长）

**用途：**
- Milvus中的结构化信息用于快速匹配
- MySQL中的完整文字支持全文检索（如需要）
- 用于相似性查找后的二次判断

**性能：**
- OCR处理时间：~100-200ms/张
- 结构化提取时间：~10-50ms/张
- 与向量化并行处理，不增加总耗时

**详细方案：** 详见 `OCR文字存储与处理方案.md`

### 4.5 检索服务模块

**功能：**
1. **向量相似度检索**
   - 对查询图片进行向量化
   - 在Milvus中搜索Top-K相似向量
   - 返回相似度分数和元数据

2. **OCR文字过滤（可选）**
   - 使用Milvus中的ocr_summary进行关键词匹配
   - 或查询MySQL进行全文检索（如需要）
   - 根据关键词进一步过滤
   - 提升检索准确率

**返回结果：**
- id, mysql_id, sku, isbn
- cover_link（图片链接）
- similarity（相似度分数）
- ocr_title, ocr_author（提取的结构化信息）
- ocr_summary（OCR摘要，用于二次判断）
- ocr_keywords（关键词，用于匹配）

**详细方案：** 详见 `OCR文字存储与处理方案.md`

## 五、技术选型说明

### 5.1 向量数据库：Milvus

**选择理由：**
- 成熟稳定，支持大规模向量检索
- 灵活的元数据存储（支持存储id、sku、isbn等）
- 支持增量插入，无需重建索引
- 阿里云提供托管服务，简化运维

**配置建议：**
- 索引类型：HNSW（性能与精度平衡）
- 相似度计算：COSINE（适合图像向量）
- 数据量支持：107万→500万可扩展

### 5.2 向量化服务：阿里云Embedding API

**选择理由：**
- 无需部署模型，开箱即用
- GPU加速，处理速度快（10-20ms/张）
- 自动处理图片预处理（尺寸调整、归一化等）
- 按调用量付费，成本可控

**模型选择：**
- 通义千问多模态模型（qwen-vl-plus）
- 或其他阿里云提供的图像向量化模型

### 5.3 OCR服务：阿里云OCR

**选择理由：**
- 高精度文字识别
- 支持多语言
- 按调用量付费
- 与向量化并行处理，不增加总耗时

### 5.4 数据源：MySQL RDS

**表结构要求：**
- 表名：可配置（默认：book_info_table）
- 必需字段：id, cover_link
- 建议字段：sku, isbn（用于业务标识）

**查询策略：**
- 分批查询，避免一次性加载大量数据
- 支持WHERE条件过滤（如只处理cover_link不为空的记录）
- 支持断点续传（通过OFFSET控制）

## 六、性能与优化

### 6.1 批量处理性能

**处理速度预估：**
- 单条处理时间：~150-250ms（向量化10-20ms + OCR 100-200ms，并行）
- 批量处理速度：~4-6条/秒（单线程）
- 并发处理（10线程）：~40-60条/秒
- 107万数据处理时间：约5-7天（10线程并发）

**优化策略：**
1. **并发处理**：使用线程池并行处理多条记录
2. **批量插入**：积累一定数量后批量插入Milvus
3. **断点续传**：支持中断恢复，避免重复处理
4. **错误重试**：API调用失败时自动重试

### 6.2 检索性能

**查询延迟：**
- 向量化时间：~10-20ms
- Milvus检索时间：< 50ms（107万数据）
- OCR过滤时间：~10-50ms（可选）
- **总延迟：< 100ms**（不含OCR过滤）

**优化策略：**
1. **索引优化**：根据数据量调整HNSW参数
2. **缓存机制**：相同图片的向量结果缓存
3. **批量查询**：支持批量图片查询（如果业务需要）

### 6.3 成本控制

**API调用成本：**
- Embedding API：按调用次数计费
- OCR API：按调用次数计费
- 107万数据处理：约214万次API调用（向量化+OCR）

**优化建议：**
1. **去重处理**：通过cover_hash避免重复处理
2. **批量API**：如果支持，优先使用批量API
3. **缓存机制**：相同图片缓存向量和OCR结果

## 八、关键设计决策

### 8.1 为什么使用Milvus而不是Hologres？

**原因：**
- 项目已确定使用Milvus
- Milvus支持灵活的元数据存储（id、sku、isbn等）
- Milvus的HNSW索引性能优异
- 阿里云提供Milvus托管服务

### 8.2 为什么并行处理向量化和OCR？

**原因：**
- OCR处理时间较长（100-200ms），与向量化并行可节省时间
- 两者互不依赖，可以同时进行
- 提升整体处理效率

### 8.3 为什么使用混合存储OCR文字？

**原因：**
- **Milvus存储关键信息**：ocr_title, ocr_author, ocr_summary, ocr_keywords
  - 检索时直接获取，无需查询MySQL
  - 支持快速关键词匹配
- **MySQL存储完整文字**（可选）
  - 无长度限制
  - 支持全文检索
  - 便于后续分析
- **兼顾性能和功能**：快速过滤 + 精确检索

### 8.4 为什么使用感知哈希去重？

**原因：**
- 快速识别完全相同的图片
- 避免重复处理，节省API调用成本
- 提升处理效率

## 九、风险与应对

### 9.1 技术风险

**风险1：API限流**
- **应对**：控制并发数，实现重试机制，使用指数退避

**风险2：图片下载失败**
- **应对**：记录失败记录，支持重试，跳过无法访问的图片

**风险3：Milvus插入失败**
- **应对**：实现事务机制，记录失败记录，支持重试

### 9.2 数据风险

**风险1：数据不一致**
- **应对**：定期校验Milvus与MySQL数据一致性

**风险2：向量质量**
- **应对**：小规模测试验证向量质量，根据效果调整

### 9.3 成本风险

**风险1：API调用费用**
- **应对**：通过去重减少调用次数，监控API使用量

**风险2：存储成本**
- **应对**：定期清理无效数据，优化存储策略
